第02章：所有权

学习目标

完成本章后，你将能够：

  • 理解 Rust 所有权机制的基本概念和设计动机
  • 掌握所有权三大原则：每个值有且仅有一个所有者、同一时间只能有一个所有者、所有者离开作用域时值被丢弃
  • 理解移动（Move）语义以及变量在移动后被失效的原因
  • 区分 Clone（深拷贝）和 Copy（栈拷贝）的适用场景
  • 理解函数调用中所有权的传递和返回方式
  • 能够编写利用所有权机制的 Rust 函数
与其他语言对比：如果你来自 Java 或 Go 背景，你可能习惯了垃圾回收（GC）自动管理内存。而在 C/C++ 中，你需要手动 malloc/free。Rust 选择了第三条路：通过所有权系统在编译时确保内存安全，既不需要 GC 也不需要手动释放——编译器会替你安排好一切。这意味着零运行时开销的内存安全保障。

1. 为什么需要所有权？

每种编程语言都需要管理内存在程序运行期间的使用方式。常见的内存管理策略包括：

  • 垃圾回收（GC）：如 Java、Go、JavaScript，运行时定期扫描不再使用的内存并自动回收。优势是编写代码简单，缺点是 GC 会带来性能开销和不可预测的暂停。
  • 手动管理：如 C/C++，程序员显式调用 malloc/free 或 new/delete。优势是完全控制，缺点是容易导致内存泄漏、悬垂指针、重复释放等问题。
  • 所有权系统：Rust 的方案——通过编译时的一组规则（所有权、借用、生命周期）来管理内存，既不需要 GC 也不需要手动释放。

所有权是 Rust 最独特的特性，它让 Rust 在没有垃圾回收的情况下保证了内存安全。理解所有权是你掌握 Rust 的关键一步。

2. 所有权三原则

Rust 的所有权系统基于三条核心规则。请牢牢记住它们：

  1. 每个值在 Rust 中都有一个变量作为它的"所有者"（owner）。
  2. 每个值同时只能有一个所有者。
  3. 当所有者离开作用域（scope）时，这个值将被丢弃（drop）。

这三条规则是理解后续所有所有权相关概念的基础。其中第 2 条直接导致了"移动（Move）"语义——当你把一个值赋给另一个变量时，所有权发生了转移。

let s1 = String::from("hello"); // s1 是 String "hello" 的所有者 { let s2 = s1; // 所有权已从 s1 移动到 s2 // s1 在此处已失效 } // s2 离开作用域，String "hello" 被自动 drop // 此时 String "hello" 的内存已被释放

注意第 2 条："每个值同时只能有一个所有者"。这意味着一旦所有权被转移，原来的所有者就不能再使用该值。这就是 Rust 防止"悬垂指针"和"双重释放"等内存错误的方式。

3. 移动（Move）

当一个变量的值被赋给另一个变量时，Rust 会执行"移动"操作——所有权从原变量转移到新变量，原变量被失效（invalidate），不能再被使用。

3.1 发生在堆上的移动

let s1 = String::from("hello"); let s2 = s1; // s1 的所有权移动到 s2 // println!("{}", s1); // 编译错误！s1 已失效 println!("{}", s2); // OK，s2 是新的所有者

String 类型的数据存储在堆上。当 let s2 = s1 时，Rust 没有复制堆上的数据（那会很慢），而是转移了所有权——只有栈上的指针、长度、容量信息被复制，而堆上的数据不动。但原变量 s1 被标记为失效，无法再访问。

这种设计与 C++ 的"浅拷贝"不同——C++ 的浅拷贝会导致两个指针指向同一块内存，然后两个析构函数都会尝试释放它（双重释放 bug）。Rust 通过所有权转移 + 原变量失效从根本上避免了这类问题。

3.2 发生在栈上的"复制"

对于只存储在栈上的类型（如整数），赋值操作是完整的拷贝，不存在移动的概念：

let x = 5; let y = x; // x 仍然是有效的！ println!("x = {}, y = {}", x, y); // OK

这是因为 i32 等类型实现了 Copy trait，赋值时会自动拷贝栈上的数据，旧变量仍然有效。后面会详细讨论 Copy 和 Clone 的区别。

注意：Rust 的移动语义与许多其他语言的行为不同。在 C++ 中，std::move 是一个需要主动调用的操作；而在 Java 和 Python 中，赋值只是复制引用，两个变量始终指向同一个对象。但在 Rust 中，let s2 = s1 始终意味着所有权转移——对于非 Copy 类型，这是隐式且不可逆的。务必记住这个关键区别！

4. Clone 与 Copy

4.1 Clone：显式深拷贝

如果你确实需要深度复制堆上的数据，可以使用 clone() 方法。这会复制包括堆数据在内的所有内容，原始变量保持有效。

let s1 = String::from("hello"); let s2 = s1.clone(); // 深拷贝：堆数据也被复制了 println!("s1 = {}, s2 = {}", s1, s2); // 两个都有效

clone() 是一个显式的操作，意味着你清楚地知道这里发生了深拷贝（可能代价较高）。在代码审查中，看到 clone() 就是在提醒你"这里复制了堆数据"。

4.2 Copy：隐式栈拷贝

某些类型实现了 Copy trait，赋值时会发生栈上的位拷贝（bitwise copy），原变量仍然有效。Copy 类型必须也实现了 Clone，并且只能用于栈上数据——即不包含堆分配的类型。

常见的 Copy 类型：

类型 是否 Copy 示例
i32, u32 等整数 是 let y = x; —— x 仍然有效
f64, f32 是 let b = a; —— a 仍然有效
bool 是 let b2 = b1; —— b1 仍然有效
char 是 let c2 = c1; —— c1 仍然有效
String 否 let s2 = s1; —— s1 失效（Move）
Vec<T> 否 let v2 = v1; —— v1 失效（Move）
// Copy 类型：赋值后原变量可用 let x: i32 = 42; let y = x; println!("x = {}, y = {}", x, y); // ✅ 两个都有效 // Non-Copy 类型：赋值导致移动 let s: String = String::from("hello"); let t = s; // println!("{}", s); // ❌ 编译错误：s 已移动
理解 Clone 与 Copy：简单来说，Copy 是隐式的、廉价的栈数据拷贝（赋值自动发生），而 Clone 是显式的、可能昂贵的深拷贝（需要调用 .clone()）。如果你来自 C++ 背景：可以把 Copy 看作"平凡可复制"（trivially copyable），把 Clone 看作"拷贝构造函数"——但 Rust 要求你必须显式调用 .clone()，绝不会隐式执行深拷贝。

5. 函数与所有权

所有权同样适用于函数调用。将值传递给函数时，所有权转移到函数的参数上；函数返回值时，所有权转移回调用方。

5.1 传递所有权给函数

fn take_ownership(s: String) { println!("{}", s); } // s 离开作用域，String 被 drop fn main() { let s = String::from("hello"); take_ownership(s); // 所有权移入函数 // println!("{}", s); // 编译错误！s 已移动 }

一旦你将值传递给函数，你就不再拥有它。如果你还希望在调用后继续使用它，有两种方式：

5.2 通过返回值恢复所有权

fn give_back(s: String) -> String { s // 所有权返回给调用方 } fn main() { let s1 = String::from("hello"); let s2 = give_back(s1); // 所有权转给函数，又通过返回值收回 println!("{}", s2); // OK }

5.3 接收并返回（模式）

如果你既需要传入值又需要传出值，可以使用"获取所有权再返还"的模式。这种模式非常常见，后面的章节会学到借用（borrowing）来避免这种繁琐的传递。

fn calculate_length(s: String) -> (String, usize) { let length = s.len(); (s, length) // 返回元组：将 String 所有权交还 } fn main() { let s1 = String::from("hello"); let (s2, len) = calculate_length(s1); println!("'{}' 的长度是 {}。", s2, len); }

这种通过返回值恢复所有权的方式虽然可行，但不够优雅。在后续章节中，你会学到用引用（Reference）和借用（Borrowing）来更简洁地解决这个问题。
