第08章：错误处理

学习目标

完成本章后，你将能够：

  • 理解 panic! 宏的使用场景与行为
  • 使用 Result<T, E> 处理可恢复错误
  • 使用 ? 运算符传播错误
  • 理解 unwrap 和 expect 的用途与风险
  • 使用 map 和 and_then 组合 Result/Option
  • 自定义错误类型并实现 Error trait
  • 综合处理多步操作中的错误
小提示：来自 Java 或 Python 的程序员习惯使用 try/catch 异常机制。Rust 没有异常！它使用 Result<T, E> 类型来编码"可能失败"的操作——这是类型系统的一部分，意味着调用者必须处理错误，不可能"忘记"捕获异常。

1. 不可恢复错误：panic!

当程序遇到无法处理的情况时，可以调用 panic! 宏。这会导致程序打印错误信息、清理栈（或直接终止），然后退出。

fn will_panic(n: i32) -> i32 { if n < 0 { panic!("negative number!") } n } // 调用时传入负数会导致程序崩溃 // will_panic(-5); // 运行时会 panic

什么时候使用 panic：

  • 示例代码或原型开发中
  • 测试中——测试失败时可以 panic
  • 当程序处于"不可能"的错误状态时（如数组越界）
注意：在库代码中应尽量避免 panic！库的调用者可能不希望程序因为库的内部错误而崩溃。优先返回 Result，让调用者决定如何处理错误。

2. 可恢复错误：Result

Result<T, E> 是一个枚举，表示操作可能成功或失败：

enum Result<T, E> { Ok(T), // 成功，包含值 T Err(E), // 失败，包含错误 E }
fn safe_divide(a: f64, b: f64) -> Result<f64, String> { if b == 0.0 { Err("division by zero".to_string()) } else { Ok(a / b) } } let result = safe_divide(10.0, 2.0); match result { Ok(value) => println!("Result: {}", value), Err(msg) => println!("Error: {}", msg), }

3. ? 运算符：错误的快捷传播

? 运算符是处理错误的语法糖：如果值是 Ok，则提取其中的值；如果是 Err，则提前返回这个错误。

fn divide_and_double(a: f64, b: f64) -> Result<f64, String> { // 如果 safe_divide 返回 Err，函数立即返回该错误 let result = safe_divide(a, b)?; Ok(result * 2.0) } // 上面的代码等价于： fn divide_and_double_verbose(a: f64, b: f64) -> Result<f64, String> { let result = match safe_divide(a, b) { Ok(v) => v, Err(e) => return Err(e), }; Ok(result * 2.0) }
理解 ? 运算符：? 是 Rust 最优雅的特性之一。在 Java 中你可能需要写 try { ... } catch (Exception e) { throw e; } 来传播异常。在 Rust 中，? 做了同样的事情，但仅凭一个字符！注意 ? 只能在返回 Result 或 Option 的函数中使用。

4. unwrap 与 expect

unwrap 和 expect 是 Result/Option 上的便捷方法：

方法Ok/Some 时Err/None 时
.unwrap()返回内部值panic!
.expect("msg")返回内部值panic! 并附带自定义消息
.unwrap_or(default)返回内部值返回默认值
.unwrap_or_else(fn)返回内部值执行闭包并返回结果
fn unwrap_or_default(input: Option<i32>) -> i32 { input.unwrap_or(0) } assert_eq!(unwrap_or_default(Some(5)), 5); assert_eq!(unwrap_or_default(None), 0);

5. map 与 and_then

map 和 and_then 是组合器（combinator），让你可以优雅地链式处理 Option/Result：

// map: 将 Ok/Some 中的值转换，Err/None 保持不变 fn parse_and_double(s: &str) -> Option<i32> { s.parse::<i32>().ok().map(|n| n * 2) } assert_eq!(parse_and_double("21"), Some(42)); // 成功解析并翻倍 assert_eq!(parse_and_double("x"), None); // 解析失败

and_then 与 map 的区别：

  • map 的闭包返回普通值（T -> U），结果被包裹在 Option<U> 中
  • and_then 的闭包返回 Option<U>（T -> Option<U>），结果不会"双重包裹"

6. 自定义错误类型

对于复杂的应用，应创建自己的错误类型。通常需要实现 Display 和 Error trait。

use std::fmt; #[derive(Debug)] enum CalcError { DivideByZero, NegativeSquareRoot, } impl fmt::Display for CalcError { fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result { match self { CalcError::DivideByZero => write!(f, "cannot divide by zero"), CalcError::NegativeSquareRoot => write!(f, "cannot sqrt negative"), } } } impl std::error::Error for CalcError {}

然后可以在函数中使用自定义错误：

fn sqrt_and_divide(a: f64, b: f64) -> Result<f64, CalcError> { if a < 0.0 { return Err(CalcError::NegativeSquareRoot); } if b == 0.0 { return Err(CalcError::DivideByZero); } Ok((a.sqrt()) / b) }

7. 综合：多错误处理

在实践中，一个函数可能需要处理多种不同类型的操作，每一步都可能失败。

// 解析多个字符串为 i32，收集结果或报告第一个错误 fn process_numbers(nums: &[&str]) -> Result<Vec<i32>, String> { nums.iter() .map(|s| { s.parse::<i32>() .map_err(|_| format!("cannot parse '{}' as i32", s)) }) .collect() // collect 遇到 Err 会直接返回第一个错误 } assert_eq!( process_numbers(&["1", "2", "3"]), Ok(vec![1, 2, 3]) ); assert!(process_numbers(&["1", "x", "3"]).is_err());
Rust 错误处理的哲学：与其他语言"抛出异常让上层处理"的哲学不同，Rust 鼓励你显式处理错误。这带来了几个好处：错误类型是函数签名的一部分（自文档化）；调用者不会被意外的异常击中；编译器确保你处理了所有错误路径。随着你使用 ?、map、and_then 等工具越来越熟练，这种"显式但优雅"的写法会让你感到很舒适。
