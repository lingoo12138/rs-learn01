第03章：借用与引用

学习目标

完成本章后，你将能够：

  • 理解 &T（不可变引用）和 &mut T（可变引用）的概念与用法
  • 掌握 Rust 的借用规则（borrowing rules）
  • 区分不可变引用与可变引用的使用场景和限制
  • 理解悬垂引用（dangling reference）及其防止机制
  • 在函数中正确使用引用作为参数
小提示：在第02章中我们学习了所有权——每个值在任意时刻只有一个所有者。但"所有权转移"有时很不方便：如果你只想读取某个值而不想获取它的所有权，每次都用完再"还回去"会很麻烦。引用（reference）正是为了解决这个问题而生——它允许你借用（borrow）一个值而不获取其所有权。

1. 什么是借用？

"借用"（borrowing）是 Rust 中的一个核心概念。当你创建一个对某个值的引用时，你并没有获得该值的所有权，只是临时借用一下。借用结束后，所有权仍然属于原来的所有者。

// 不使用引用——所有权转移了 fn take_ownership(s: String) { // s 获得了所有权，函数结束后 s 被释放 } // 使用引用——只是借用 fn borrow_it(s: &String) { // s 是一个引用，不拥有 String，函数结束后原值依然有效 }

借用的核心思想：引用不获取所有权，只是借用。这类似于你去图书馆借书——你拿到书（引用），可以阅读它，但书的所有权仍然属于图书馆。归还后，图书馆（原所有者）仍然拥有这本书。

注意：引用默认也是不可变的。如果你需要修改借用的值，必须使用可变引用 &mut T。这是 Rust 安全性设计的一部分，确保在没有显式声明可变的情况下，没有任何代码能够意外修改数据。

2. 不可变引用 &T

不可变引用（immutable reference）允许你读取一个值，但不允许修改它。使用 & 操作符创建不可变引用。

let x = 42; let r = &x; // 创建对 x 的不可变引用 println!("r = {}", r); // 通过引用读取值 // *r = 43; // 编译错误！不可变引用不能修改值

2.1 多个不可变引用可以同时存在

不可变引用是共享的——同一时刻可以有多个不可变引用指向同一个值，因为读取操作不会造成数据竞争。

let x = 42; let r1 = &x; let r2 = &x; // 多个不可变引用，没问题 let r3 = &x; // 还可以更多 println!("{}, {}, {}", r1, r2, r3); // 同时使用所有引用
类比：多个同学可以同时借阅同一本书的阅读权限，大家都只能阅读，不能修改书中的内容。图书馆（Rust 编译器）允许无限多人同时阅读同一本书，因为阅读不会产生冲突。

3. 可变引用 &mut T

可变引用（mutable reference）允许你读取和修改一个值。使用 &mut 操作符创建可变引用。

let mut x = 42; let r = &mut x; // 创建对 x 的可变引用 *r += 1; // 通过可变引用修改值——使用 * 解引用 println!("x = {}", x); // x = 43

3.1 同一时刻只能有一个可变引用

可变引用是独占的——同一时刻只能有一个可变引用指向同一个值。这防止了数据竞争（data race）。

let mut x = 42; let r1 = &mut x; // let r2 = &mut x; // 编译错误！不能同时创建两个可变引用 let r1 = &mut x; // 前一个 r1 已不再使用，可以创建新的 *r1 += 1;
注意：可变引用的独占性不仅体现在同一作用域中不能同时存在两个可变引用，还体现在——当你有一个可变引用时，也不能同时有任何不可变引用。这是因为不可变引用假设值不会变化，而可变引用可能正在修改它，两者同时存在会导致不一致。

4. 借用规则

Rust 的借用规则总结如下，这是 Rust 内存安全的核心保障之一：

规则 说明
规则一 同一时刻，只能有一个可变引用（&mut T），或者多个不可变引用（&T），不能同时存在
规则二 引用必须始终有效——引用的生命周期不能超过被引用值的生命周期
规则三 引用不会获取所有权——借用的值在引用消失后仍然存在（除非值被移动或销毁）

简单记忆：要么一个可变引用，要么多个不可变引用，不能同时有两者。

// 正确：多个不可变引用 let x = 42; let r1 = &x; let r2 = &x; println!("{} {}", r1, r2); // 正确：一个可变引用（没有其他引用） let mut y = 10; let r = &mut y; *r += 1; // 错误：不能同时拥有可变引用和不可变引用 // let mut z = 42; // let r1 = &z; // let r2 = &mut z; // println!("{}", r1); // 编译错误！

4.1 引用的作用域

引用的作用域从它创建的地方开始，到最后一次使用的地方结束。这被称为 NLL（Non-Lexical Lifetimes，非词法生命周期），是 Rust 编译器的智能特性。

let mut x = 42; let r = &x; // r 的作用域开始 println!("{}", r); // r 的最后一次使用 let r2 = &mut x; // 没问题！r 已经在上一行被"释放" *r2 += 1;

如果引用的最后一次使用之后还有可变引用，编译器不会报错，因为引用已经不再被使用。这种设计让代码更灵活，同时保持安全性。

理解 NLL：在早期的 Rust 版本中（2018 之前），引用的作用域从创建到花括号结束，这导致许多合理的代码模式无法通过编译。Rust 团队推出 NLL 后，引用的作用域精确到最后一次使用的位置，大大改善了编程体验。这是 Rust 在"严格"和"实用"之间平衡的一个典型例子。

5. 悬垂引用

悬垂引用（dangling reference）指的是指向已经被释放的内存的引用。在 C 或 C++ 中，悬垂指针是常见的 bug 来源。Rust 编译器在编译时就能检测并阻止悬垂引用的产生。

// 以下代码无法编译——Rust 会阻止悬垂引用 // fn dangling() -> &String { // let s = String::from("hello"); // &s // s 在函数结束时被释放，返回的引用将指向无效内存 // } // 编译错误：missing lifetime specifier

Rust 通过生命周期（lifetime）机制来确保引用始终有效。生命周期标注告诉编译器不同引用之间的关系，帮助编译器在编译时验证引用的有效性。我们将在第06章中深入学习生命周期。

// 正确的方式：将值返回，所有权随返回值转移 fn no_dangling() -> String { let s = String::from("hello"); s // 所有权转移给调用者 }
关键理解：悬垂引用问题的本质是"引用和它所引用的值的生命周期不匹配"。如果被引用的值已经被释放，而引用仍然存在，就产生了悬垂引用。Rust 在编译时通过生命周期检查来杜绝这类问题，确保引用永远比它所引用的值"活得更短"。这正是 Rust 内存安全"零成本抽象"的体现——所有检查都在编译期完成，运行时没有任何开销。

练习题

打开 exercises/src/chapter03_borrow/mod.rs 文件，找到对应的函数填空，然后运行 cargo test 验证你的答案。

  1. 不可变引用 — 创建对 x 的不可变引用 r，通过解引用返回 *r 的值。
    提示：使用 &x 创建引用，用 *r 解引用获取值。
  2. 可变引用与修改 — 创建对 s 的可变引用，调用 push_str 追加字符串。
    提示：先用 &mut s 创建可变引用，再调用 r.push_str(" world")。
  3. 多个不可变引用 — 创建两个不可变引用 r1 和 r2 指向 x，返回两者之和。
    提示：多个不可变引用可以共存，用 *r1 + *r2 计算结果。
  4. 引用作为参数 — 实现 ref_length 函数，通过引用计算 String 的长度。
    提示：参数类型为 &String，调用 s.len() 即可。
  5. 可变引用参数 — 实现 add_forty 函数，通过可变引用给参数加 40。
    提示：参数类型为 &mut i32，使用 *num += 40 修改值。
  6. 字符串切片与引用 — 实现 first_char 函数，返回字符串的第一个字符。
    提示：参数类型为 &str，使用 s.chars().next().unwrap()。
  7. 理解错误 — 解释为什么同时拥有可变和不可变引用会导致编译错误。
    提示：这是 Rust 借用规则的核心——要么一个可变引用，要么多个不可变引用。
  8. 交换值 — 实现 swap 函数，交换两个 i32 变量的值。
    提示：使用两个 &mut i32 参数，用临时变量交换。
